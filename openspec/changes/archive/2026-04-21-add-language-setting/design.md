## Context

当前项目使用 next-intl 进行国际化，支持中文（zh）和英文（en）两种语言。语言切换组件（LanguageSwitcher）位于顶部导航栏，包含：
- 下拉菜单显示当前语言
- 切换确认弹窗（提醒用户切换语言会影响提示词模板和输出语言）
- 路由跳转逻辑（使用 next-intl 的路由切换）

设置中心（Profile 页面）目前包含两个分区：
- API 配置（apiConfig）
- 扣费记录（billing）

## Goals / Non-Goals

**Goals:**
- 在设置中心添加"通用设置"分区，包含语言切换功能
- 从导航栏移除 LanguageSwitcher 组件
- 保持现有的语言切换逻辑和确认弹窗
- 语言默认值保持为中文（zh）

**Non-Goals:**
- 不修改 i18n 核心配置或路由逻辑
- 不添加其他语言选项
- 不修改语言切换确认弹窗的文案或行为

## Decisions

### 1. 复用现有 LanguageSwitcher 组件

**决定**：直接将现有的 `LanguageSwitcher` 组件移至设置中心，而非创建新组件。

**理由**：
- 组件已包含完整的语言切换逻辑和确认弹窗
- 避免代码重复
- 减少测试工作量

**替代方案**：
- 创建新的设置专用语言切换组件 → 增加维护成本
- 将语言切换逻辑提取为 hook → 当前组件逻辑简单，过度抽象

### 2. 设置页面布局调整

**决定**：在左侧导航菜单中添加"通用设置"选项，点击后显示语言切换界面。

**理由**：
- 与现有的"API 配置"和"扣费记录"分区保持一致的交互模式
- 为未来添加其他通用设置预留扩展空间

### 3. 导航栏简化

**决定**：完全移除导航栏中的 LanguageSwitcher，不保留任何语言指示器。

**理由**：
- 大多数用户使用中文，无需频繁切换
- 简化导航栏视觉
- 用户可通过设置中心随时切换语言

## Risks / Trade-offs

**风险 1：用户可能找不到语言切换入口**
→ 缓解：在首次使用或迁移时，可通过更新提示告知用户语言设置已移至设置中心

**风险 2：设置页面布局变更可能影响现有用户习惯**
→ 缓解：保持"API 配置"为默认选中分区，用户无需频繁访问"通用设置"

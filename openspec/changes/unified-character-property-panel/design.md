## Context

### 当前状态

系统存在两套角色属性编辑机制：

1. **生成前编辑** (`CharacterProfileCard` + `CharacterProfileDialog`)
   - 用于 AI 选角完成后的档案确认
   - 可编辑：角色层级、原型、性格标签、服装华丽度、建议色彩、辨识标志、视觉关键词
   - 只读展示：性别、年龄、时代、阶层、身份
   - 操作：编辑档案 → 确认并生成

2. **生成后编辑** (`CharacterEditModal` - 基本属性区域)
   - 用于已有形象的角色编辑
   - 可编辑：性别、年龄、身份、身高、外貌体型、性格标签
   - 不可编辑：时代、阶层、服装华丽度、建议色彩等

### 数据结构

`CharacterProfileData` 类型包含完整的角色属性：
- `role_level`: 角色重要性层级 (S/A/B/C/D)
- `archetype`: 角色原型
- `personality_tags`: 性格标签
- `era_period`: 时代背景
- `social_class`: 社会阶层
- `identity`: 身份
- `height`: 身高
- `body_type`: 外貌体型
- `costume_tier`: 服装华丽度 (1-5)
- `suggested_colors`: 建议色彩
- `primary_identifier`: 主要辨识标志
- `visual_keywords`: 视觉关键词
- `gender`: 性别
- `age_range`: 年龄段

### 约束条件

- 需要兼容现有的 `mode: 'asset-hub' | 'project'` 双模式
- 需要支持生成前后两种状态
- 需要保持与现有 API 的兼容性

## Goals / Non-Goals

**Goals:**
- 创建统一的角色属性编辑组件 `UnifiedCharacterPropertyPanel`
- 合并 `CharacterProfileData` 的所有属性编辑功能
- 自动生成并展示文生图 AI 提示词
- 自动生成并展示音色 AI 提示词
- 支持一键生成角色形象和音色

**Non-Goals:**
- 不修改后端 API 结构
- 不改变现有的角色生成流程逻辑
- 不修改 `CharacterProfileData` 类型定义
- 不影响 Asset Hub 和 Project 的双模式架构

## Decisions

### Decision 1: 组件架构

**选择**: 创建新的统一组件 `UnifiedCharacterPropertyPanel`，替代现有的 `CharacterProfileDialog` 并整合到 `CharacterEditModal` 中。

**理由**:
- 避免直接修改现有组件导致的回归风险
- 新组件可以更好地组织 UI 布局
- 渐进式迁移，保留旧组件作为回退

**替代方案**:
- 方案 A: 直接扩展 `CharacterEditModal` → 风险高，改动大
- 方案 B: 创建两个独立面板 → 用户仍需在两个界面切换

### Decision 2: 属性分组

**选择**: 将属性分为四个逻辑区域：
1. **基础信息**: 性别、年龄、身份、身高、外貌体型
2. **角色定位**: 角色层级、原型、性格标签
3. **视觉设定**: 时代、阶层、服装华丽度、建议色彩、辨识标志、视觉关键词
4. **AI 提示词**: 文生图提示词、音色提示词（只读展示）

**理由**:
- 符合用户认知流程：先确认基础信息，再设定视觉细节
- AI 提示词作为最后确认环节，方便用户检查

### Decision 3: 提示词生成策略

**选择**: 实时生成提示词，用户每次修改属性后自动更新提示词预览。

**理由**:
- 用户可以即时看到修改对提示词的影响
- 减少手动触发生成的操作步骤

**实现方式**:
- 文生图提示词：调用现有的 `/api/.../generate-character-prompt` API
- 音色提示词：基于属性组合生成模板化提示词

### Decision 4: 生成按钮行为

**选择**: 单一"确认并生成"按钮，根据当前状态决定行为：
- **生成前**: 保存属性 → 生成形象 → 生成音色
- **生成后**: 保存属性 → 重新生成形象（如需要） → 重新生成音色（如需要）

**理由**:
- 简化用户操作，无需区分"仅保存"和"保存并生成"
- 保持与现有 `saveAndGenerate` 行为一致

## Risks / Trade-offs

### Risk 1: 组件复杂度增加
**风险**: 统一组件需要处理两种模式（asset-hub/project）和两种状态（生成前/后），逻辑复杂度上升
**缓解**: 使用状态机模式管理组件状态，编写充分的单元测试

### Risk 2: 提示词生成延迟
**风险**: 实时生成提示词可能导致 UI 卡顿
**缓解**: 使用防抖（debounce）处理属性变更，添加 loading 状态

### Risk 3: 向后兼容性
**风险**: 替换现有组件可能影响依赖这些组件的其他功能
**缓解**: 保留旧组件，新组件通过 feature flag 控制，渐进式迁移

### Trade-off: 属性可编辑性
**权衡**: 是否允许生成后编辑所有属性？
- 选择：允许，但需要用户确认重新生成
- 理由：用户可能发现设置错误需要修正

## Open Questions

1. **音色生成是否与形象生成强绑定？**
   - 当前设计假设同时生成，但可能需要支持只生成其中一个

2. **属性变更后是否自动标记为需要重新生成？**
   - 需要定义哪些属性变更触发重新生成

3. **提示词编辑权限？**
   - AI 生成的提示词是否允许用户手动编辑？
